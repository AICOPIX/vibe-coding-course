1.4 前端、后端、数据库、API、框架，一次看懂

这 5 个词不是不同等级，而是同一个项目里的不同职责和组织方式。先把它们放回项目地图里，后面很多概念都会顺。

增强版 概念课

本节目标

  • 先分清前端、后端、数据库、API、框架各自到底负责什么。
  • 理解它们为什么会在项目变复杂时逐步出现，而不是一开始就必须全部上齐。
  • 学会把一个新需求先判断到正确层，再决定下一步怎么做。

为什么会有这些概念

一个很简单的静态页面，可能只需要页面文件和样式就够了。但项目一旦开始有交互、数据、权限、搜索、持续更新和多人协作，就会出现新的问题：谁负责展示、谁负责规则、谁负责保存、谁负责通信、谁负责让项目别越做越乱。

这些概念不是为了把初学者绕晕，而是为了把复杂项目拆成能理解、能协作、能维护的结构。

5 个词分别在解决什么问题

  • 前端：让用户看到内容、完成点击输入，并及时得到反馈。
  • 后端：处理业务逻辑、权限、流程，以及数据读写的组织。
  • 数据库：把用户资料、评论、进度、课程内容等信息长期保存起来。
  • API：给页面和系统之间提供一个稳定交换数据与能力的边界。
  • 框架：帮你统一页面路由、组件组织、数据获取和工程结构。

一张最小项目地图

  1. 用户在前端页面上浏览、点击、输入。
  2. 前端通过 API 发出请求。
  3. 后端接收请求并处理业务逻辑。
  4. 后端按需要读写数据库。
  5. 后端再把结果通过 API 返回给前端。
  6. 框架负责把页面、路由、组件和数据获取组织得更稳定。

先记住这条链，后面你再听到报错、接口、持久化、框架这些词，就更容易定位到项目里的位置。

一个贯穿式 mini 案例

把当前这套静态课程站，想象成未来要升级成“可搜索、可登录、可管理内容”的课程平台。

  • 前端：课程页面、搜索框、登录界面、个人中心。
  • 后端：处理登录、权限判断、搜索规则和内容管理流程。
  • 数据库：保存用户、课程、评论、学习记录。
  • API：让前端去请求登录、搜索、评论和进度等能力。
  • 框架：帮助你组织页面路由、组件复用和整体工程结构。

这个案例最重要的判断是：不是所有项目一开始都需要五样全上，而是项目一复杂，这几层就会逐步出现。

在当前项目里的位置

你现在这个仓库，最明显的是内容源、前端页面和静态发布流程，所以它目前更接近“以静态内容为主的前端项目”。但如果后面要加搜索、评论、后台管理、用户系统，后端、数据库、API、框架就会真正进入项目决策。

初学者最容易误解什么

  • 误以为前端比后端简单。实际上，两者只是职责不同，复杂度来源也不同。
  • 误以为 API 只属于后端。对前端来说，API 恰恰是它连接数据能力的桥。
  • 误以为数据库是一个“大黑盒”。先把它理解成项目长期保存和查询信息的地方就够了。
  • 误以为框架必须先精通，才能开始做项目。更合理的顺序是先建立项目地图，再决定何时引入框架。
  • 误以为做一个网页时，这 5 个词一定要同时出现。真正的判断重点是：当前版本到底需要哪几层。

本节练习

先看当前课程站，分别写一句话说明：现在哪些部分最明显属于前端；为什么暂时不一定需要完整后端；如果以后要保存学习进度，数据库会负责什么；如果页面要读取进度，API 会扮演什么角色。
