## AI 编程方式在项目里的位置

如果把 AI 编程看成一条连续光谱，大致可以分成下面几类：

- AI 补全：你主写，AI 帮你补代码
- 对话助手：你提问，AI 解释、排错、改写
- AI IDE：你在编辑器里直接协作生成、修改、重构
- 应用生成器：你描述目标，AI 帮你快速搭出原型
- 胶水编程：你已有页面、组件、API 或服务，AI 帮你把它们连接起来

对当前这个课程项目来说，最实用的路线通常不是“纯手写代码”，而是：

1. 先用自然语言说明目标
2. 再让 AI 生成页面或结构
3. 最后用胶水编程把已有模块、接口和服务接起来

所以，胶水编程不是离 Vibe Coding 很远的另一套东西，而是项目进入真实落地阶段后经常会碰到的能力。

## 什么是胶水编程？

### 官方定义

胶水代码（Glue Code）是指连接两个或多个原本不兼容的软件组件的代码。它本身不执行主要业务逻辑，而是作为桥梁让不同系统能够协同工作。

参考来源：
- Wikipedia: Glue code
- Python Wiki: Glue Language

### 生活化理解

可以把胶水编程理解成“转接头”。

- 你有一个页面
- 你有一个 AI 接口
- 你有一个数据库
- 你可能还要接一个支付、地图或登录服务

它们各自能工作，但未必天然能配合。胶水代码的作用，就是把这些东西接起来，让整个流程跑通。

## 胶水编程的核心概念

### 1. 适配器模式

最常见的胶水代码形式，就是把一个接口转换成另一个接口。

例如：

- 系统 A 输出 `{ name, age }`
- 系统 B 需要 `{ userName, userAge }`

这时候中间的转换层，就是胶水代码。

### 2. 桥接模式

连接两个独立变化的维度，让它们可以组合使用。

例如：

- 多种支付方式
- 多种货币或订单场景

胶水代码负责让这些能力可以自由组合，而不是每次都重写一套流程。

### 3. 外观模式

把复杂子系统包装成一个简单入口。

例如“我要下单”这一个动作，背后其实要经过：

- 库存系统
- 支付系统
- 物流系统

用户不需要看见这些复杂度，胶水代码把它们包成一个更容易调用的接口。

## 胶水编程的常见方法

### 方法1：API 调用与封装

最常见的场景就是调用第三方 API，并整理成自己页面能使用的格式。

例如：

- 调用天气 API
- 取出温度和天气状态
- 转成页面右上角的小组件

### 方法2：数据格式转换

不同系统的数据格式经常不一样，胶水代码负责在中间转换。

常见转换包括：

- JSON 和 XML
- CSV 和 JSON
- 数据库记录和前端对象
- 旧字段名和新字段名

### 方法3：事件监听与转发

监听一个系统的事件，再触发另一个系统动作。

例如：

- 用户下单成功
- 触发短信通知
- 触发邮件提醒
- 更新 CRM 或后台面板

### 方法4：错误处理与重试

两个系统一旦连起来，失败场景就会增多。

例如：

- 网络超时
- API 限流
- 第三方服务暂时不可用

胶水代码通常还要负责重试、兜底、记录日志和提示用户。

## 常见的“积木块”

胶水编程很少是从零造轮子，它更像是在组装现成能力。

### 第三方服务

- 支付服务：微信支付、支付宝、Stripe、PayPal
- 通讯服务：邮件、短信、推送
- 地图服务：高德、百度、腾讯地图、Google Maps
- 身份认证：GitHub 登录、Google 登录、微信登录
- 云存储：OSS、COS、S3
- AI 服务：OpenAI、Claude、通义千问等

### 开源组件

- UI 组件库
- 图表库
- 富文本编辑器
- 日期时间处理库
- 文件上传组件
- 通知提示组件

项目变复杂后，你会发现自己并不是一直“写功能”，更多时候是在“让不同功能一起工作”。

## 胶水编程 vs Vibe Coding

| 对比项 | Vibe Coding | 胶水编程 |
|--------|-------------|----------|
| 起点 | 从零描述想法 | 已有组件、服务、页面 |
| 目标 | 快速生成功能 | 连接现有系统 |
| 你的角色 | 说明想做什么 | 说明要怎么连接 |
| AI 的角色 | 生成主体代码 | 生成连接代码 |
| 适合场景 | 原型、独立功能、新页面 | 集成、扩展、联动 |

它们不是对立关系，而是常常前后衔接：

- 前期用 Vibe Coding 搭出页面和功能雏形
- 中后期用胶水编程接入登录、数据库、支付、AI、部署流程

## 什么时候会用到胶水编程？

### 场景1：项目功能扩展

一开始你只做了一个静态页面，后来想加：

- 联系表单
- 评论系统
- 邮件通知
- AI 助手

这时候大多不是重写网站，而是把现有页面和新服务接起来。

### 场景2：使用外部数据

例如做天气、地图、资讯、股票、翻译、AI 问答等功能时，数据往往都来自外部服务。

### 场景3：系统集成

一个真实项目里可能会同时存在：

- 前端页面
- 后端接口
- 数据库
- 第三方登录
- 统计工具

胶水编程就是让这些东西变成一个整体。

### 场景4：旧系统升级

当旧页面、旧接口不能直接重做时，胶水层往往是最现实的过渡方案。

## 常见挑战

- 胶水代码散落在各处，后期难维护
- 第三方 API 更新后，旧接法失效
- 错误处理复杂，尤其是多系统联动
- 外部调用太多，性能和稳定性下降

所以胶水代码虽然看起来“不显眼”，但在项目里往往很关键。

## 给设计师的建议

你不需要一开始就精通所有接口细节，但你最好能做到：

- 知道有哪些可复用的服务和组件
- 能描述清楚 A 和 B 要怎么配合
- 能判断联动结果对不对
- 知道哪里是页面问题，哪里是接口问题

这会让你和 AI 配合时效率高很多。
