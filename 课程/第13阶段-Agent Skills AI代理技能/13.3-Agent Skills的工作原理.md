## 🎯 核心原理

### 简单流程

```
你的需求
    ↓
Copilot 分析
    ↓
匹配相关 Skill
    ↓
加载 Skill 指导
    ↓
执行任务
    ↓
完成
```

## 📊 三层加载系统

Agent Skills 使用"渐进式加载"来高效管理上下文。这是一个聪明的设计，确保即使安装了很多 Skills，也不会浪费 AI 的"思考空间"。

### 第 1 层：发现（Discovery）

**什么被加载**：
- Skill 的名称
- Skill 的描述
- 这些信息来自 SKILL.md 的 YAML 前置信息

**何时加载**：
- 总是加载（当 Copilot 启动时）

**消耗的上下文**：
- 很少（只是元数据）

**Copilot 的思考过程**：
```
我有这些 Skills 可用：
1. react-component-creation - 创建 React 组件
2. testing-guide - 编写测试
3. api-integration - 集成 API
4. ...

用户要求：创建一个 React 组件

匹配的 Skill：react-component-creation ✓
```

### 第 2 层：加载指导（Instructions Loading）

**什么被加载**：
- SKILL.md 文件的完整内容
- 所有的指导、步骤、最佳实践

**何时加载**：
- 当用户的需求匹配某个 Skill 的描述时

**消耗的上下文**：
- 中等（取决于 SKILL.md 的长度）

**Copilot 的思考过程**：
```
用户要求：创建一个 React 组件

加载 react-component-creation Skill 的完整指导...

现在我知道：
- 应该使用 TypeScript
- 应该使用 Tailwind CSS
- 应该遵循的最佳实践
- 参考的代码示例

按照这些指导创建组件...
```

### 第 3 层：访问资源（Resource Access）

**什么被加载**：
- Skill 文件夹中的其他文件
- 脚本、模板、配置示例等

**何时加载**：
- 只有当 Copilot 需要时才加载
- 例如，当 SKILL.md 中引用了某个文件时

**消耗的上下文**：
- 最少（只加载需要的文件）

**Copilot 的思考过程**：
```
SKILL.md 中说：参考 [测试模板](./test-template.js)

加载 test-template.js 文件...

现在我可以看到具体的代码模板，
按照这个模板创建测试...
```

## 🔄 完整工作流程

### 场景：创建一个 React 组件

#### 步骤 1：你提出需求

```
你：帮我创建一个按钮组件，
    使用 TypeScript 和 Tailwind CSS
```

#### 步骤 2：Copilot 分析（第 1 层）

```
Copilot 检查可用的 Skills：
- react-component-creation ✓ 匹配！
- testing-guide ✗ 不相关
- api-integration ✗ 不相关

决定：加载 react-component-creation Skill
```

#### 步骤 3：加载指导（第 2 层）

```
加载 SKILL.md 内容：

# React 组件创建指南

## 何时使用此 Skill
- 创建新的 React 组件
- 确保组件遵循最佳实践
- 使用 TypeScript 进行类型检查
- 使用 Tailwind CSS 进行样式设计

## 组件结构
interface ComponentProps {
  // 定义 Props 类型
}

export const MyComponent: React.FC<ComponentProps> = ({
  // Props
}) => {
  return (
    <div>
      {/* 组件内容 */}
    </div>
  );
};

## 最佳实践
- ✅ 使用 TypeScript 进行类型检查
- ✅ 为 Props 添加默认值
- ✅ 使用 Tailwind CSS 的响应式类
...
```

#### 步骤 4：访问资源（第 3 层）

```
SKILL.md 中提到了 [示例](./example.tsx)

加载 example.tsx 文件...

现在 Copilot 有了完整的参考
```

#### 步骤 5：执行任务

```
Copilot 按照 Skill 的指导创建组件：

✓ 使用 TypeScript 定义 Props
✓ 使用 React.FC 类型
✓ 使用 Tailwind CSS 类名
✓ 遵循最佳实践
✓ 包含 JSDoc 注释
```

#### 步骤 6：返回结果

```
Copilot 生成的代码：

interface ButtonProps {
  label: string;
  variant?: 'primary' | 'secondary';
  onClick?: () => void;
}

export const Button: React.FC<ButtonProps> = ({
  label,
  variant = 'primary',
  onClick,
}) => {
  const baseStyles = 'px-4 py-2 rounded font-semibold';
  const variantStyles = {
    primary: 'bg-blue-500 text-white hover:bg-blue-600',
    secondary: 'bg-gray-200 text-gray-800 hover:bg-gray-300',
  };

  return (
    <button
      className={`${baseStyles} ${variantStyles[variant]}`}
      onClick={onClick}
    >
      {label}
    </button>
  );
};
```

## 🎯 Copilot 如何决定使用哪个 Skill

### 匹配算法

Copilot 使用以下因素来决定是否使用某个 Skill：

1. **关键词匹配**
   ```
   你的需求中的关键词 vs Skill 描述中的关键词
   
   例子：
   你说："创建 React 组件"
   Skill 描述："使用 TypeScript 和 Tailwind CSS 创建 React 组件"
   匹配度：高 ✓
   ```

2. **语义相似性**
   ```
   Copilot 理解你的意思，而不仅仅是字面匹配
   
   例子：
   你说："我需要一个按钮"
   Skill 描述："创建 React 组件"
   匹配度：中等 ✓（按钮是一种组件）
   ```

3. **上下文**
   ```
   Copilot 考虑你正在做什么
   
   例子：
   你在 React 项目中
   你说："创建一个组件"
   Skill 描述："创建 React 组件"
   匹配度：高 ✓
   ```

### 多个 Skill 匹配的情况

如果多个 Skill 都匹配，Copilot 会：

1. 选择最相关的 Skill
2. 有时会加载多个 Skill 的指导
3. 综合使用多个 Skill 的最佳实践

## 💡 最佳实践

### 1. 写清楚的描述

```markdown
❌ 不好：
description: 编程相关

✅ 好：
description: 使用 TypeScript 和 Jest 创建单元测试的指南。
            当你需要为 React 组件编写测试时使用此 Skill。
```

**为什么**：清楚的描述帮助 Copilot 准确匹配你的需求。

### 2. 组织清晰的结构

```markdown
✅ 好的结构：

# Skill 名称

## 何时使用此 Skill
- 场景 1
- 场景 2

## 步骤
1. 步骤 1
2. 步骤 2

## 最佳实践
- 实践 1
- 实践 2

## 示例
[参考示例](./example.js)
```

**为什么**：清晰的结构让 Copilot 更容易理解和应用指导。

### 3. 包含具体的代码示例

```markdown
✅ 好：
## 示例

```typescript
interface Props {
  name: string;
}

export const Component: React.FC<Props> = ({ name }) => {
  return <div>{name}</div>;
};
```

❌ 不好：
## 示例
创建一个组件
```

**为什么**：具体的代码示例让 Copilot 知道你期望的确切格式。

### 4. 链接到参考文件

```markdown
✅ 好：
查看 [测试模板](./test-template.js) 了解如何编写测试。

❌ 不好：
查看测试模板文件了解如何编写测试。
```

**为什么**：明确的链接让 Copilot 知道在哪里找到参考资源。

## 🔍 调试 Skills

### 如何知道 Skill 是否被加载

1. **查看 Copilot 的响应**
   ```
   如果 Copilot 的代码遵循了你的 Skill 指导，
   说明 Skill 被加载了
   ```

2. **检查 Copilot Chat 的上下文**
   ```
   在 VS Code 中，你可以看到 Copilot 使用了哪些文件
   ```

### 常见问题

#### 问题 1：Skill 没有被加载

**可能的原因**：
- 描述不够清楚
- 关键词不匹配
- Skill 文件夹位置不对

**解决方案**：
- 改进 Skill 的描述
- 添加更多相关的关键词
- 检查文件夹位置（`.github/skills/` 或 `~/.copilot/skills/`）

#### 问题 2：Skill 被加载了，但没有被正确应用

**可能的原因**：
- 指导不够具体
- 缺少代码示例
- 最佳实践不清楚

**解决方案**：
- 添加更多具体的步骤
- 包含完整的代码示例
- 明确列出最佳实践

## 📚 理解上下文管理

### 为什么三层加载很重要

想象你有 100 个 Skills：

#### 没有三层加载（低效）
```
加载所有 100 个 Skills 的完整内容
→ 消耗大量上下文
→ Copilot 的"思考空间"被占满
→ 响应变慢
→ 质量下降
```

#### 有三层加载（高效）
```
第 1 层：只加载 100 个 Skills 的名称和描述
        （很小的文件）

第 2 层：当需要时，加载相关 Skill 的完整内容
        （只加载需要的）

第 3 层：当需要时，加载参考文件
        （按需加载）

→ 上下文使用高效
→ 响应快速
→ 质量高
```

## 🎓 关键要点

✅ **三层加载系统**
- 第 1 层：发现（总是加载）
- 第 2 层：指导（按需加载）
- 第 3 层：资源（按需加载）

✅ **Copilot 的决策过程**
- 分析你的需求
- 匹配相关 Skills
- 加载相关指导
- 执行任务

✅ **最佳实践**
- 写清楚的描述
- 组织清晰的结构
- 包含具体的代码示例
- 链接到参考文件
