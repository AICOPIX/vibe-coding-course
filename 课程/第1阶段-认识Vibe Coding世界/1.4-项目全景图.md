# 1.4 项目全景图：AI 编程中的角色分工

## 🎯 学习目标

- 了解一个项目从头到尾涉及哪些层面
- 明白每个层面的重要性排序
- 知道 AI 编程时你该关注什么、AI 负责什么

---

## 📖 为什么要学这个？

很多人开始 AI 编程时会困惑：
- "我到底要懂多少技术？"
- "哪些事情我必须自己做？"
- "哪些可以完全交给 AI？"

这节课帮你建立**全局视角**，让你知道自己在整个项目中的位置。

---

## 🏗️ 项目的完整层级

一个项目从想法到上线，要经过这些层面：

```
┌─────────────────────────────────────────────────────────┐
│  1. 目标/愿景    为什么要做这个？解决什么问题？           │  ← 最抽象
├─────────────────────────────────────────────────────────┤
│  2. 需求/功能    具体要做哪些功能？给谁用？              │
├─────────────────────────────────────────────────────────┤
│  3. 用户体验     用户怎么用？流程是什么？                │
├─────────────────────────────────────────────────────────┤
│  4. UI 设计      长什么样？布局、颜色、字体              │
├─────────────────────────────────────────────────────────┤
│  5. 架构设计     系统怎么组织？模块怎么划分？            │
├─────────────────────────────────────────────────────────┤
│  6. 技术栈       用什么工具/框架/语言？                  │
├─────────────────────────────────────────────────────────┤
│  7. 数据设计     存什么数据？数据之间什么关系？          │
├─────────────────────────────────────────────────────────┤
│  8. 代码实现     具体的代码怎么写                        │
├─────────────────────────────────────────────────────────┤
│  9. 测试验收     怎么验证做对了？                        │
├─────────────────────────────────────────────────────────┤
│ 10. 部署上线     怎么让别人能访问？                      │
├─────────────────────────────────────────────────────────┤
│ 11. 运维监控     上线后怎么维护？出问题怎么发现？        │  ← 最具体
└─────────────────────────────────────────────────────────┘
```


---

## 🏠 用盖房子来理解

| 层级 | 盖房子比喻 | 说明 |
|------|-----------|------|
| 目标/愿景 | 为什么要盖房子？自住还是出租？ | 最根本的"为什么" |
| 需求/功能 | 要几个房间？要不要车库？ | 具体要什么 |
| 用户体验 | 进门先到哪？动线怎么走？ | 怎么用起来顺手 |
| UI 设计 | 装修风格、颜色、家具 | 长什么样 |
| 架构设计 | 承重墙、水电管道布局 | 内部结构 |
| 技术栈 | 用什么材料？砖头还是木头？ | 用什么工具 |
| 数据设计 | 水表电表怎么装？ | 信息怎么存 |
| 代码实现 | 砌每一块砖 | 具体动手做 |
| 测试验收 | 验收检查、试住 | 检查做对没 |
| 部署上线 | 搬家入住 | 正式使用 |
| 运维监控 | 日常维护、水电检修 | 持续维护 |

---

## ⚖️ 重要性排序

**越上层越重要，改错的代价越大：**

| 层级 | 改错的代价 | 比喻 |
|------|-----------|------|
| 目标错了 | 💀 全部白做 | 盖完发现根本不需要这个房子 |
| 需求错了 | 😱 推倒重来 | 盖了3层发现只需要1层 |
| 体验错了 | 😰 大改 | 动线不对，要拆墙重新规划 |
| 设计错了 | 😓 中改 | 装修风格不对，要重新装修 |
| 架构错了 | 😥 中改 | 承重墙位置错了，要加固 |
| 技术栈错了 | 😅 小改 | 换个牌子的材料 |
| 代码错了 | 😊 微调 | 某块砖砌歪了，敲掉重砌 |

**结论：先想清楚"做什么"，再让 AI 帮你"怎么做"**

## 👥 AI 编程时的分工

这是最重要的部分！

```
┌────────────────────┬──────────────┬─────────────────────┐
│       层级         │    谁负责     │      你要做什么      │
├────────────────────┼──────────────┼─────────────────────┤
│ 1. 目标/愿景       │    你 100%   │ 想清楚为什么做       │
│ 2. 需求/功能       │    你 100%   │ 列出要做的功能       │
│ 3. 用户体验        │    你 80%    │ 画流程图/描述流程    │
│ 4. UI 设计         │    你 90%    │ 你是设计师，强项！   │
├────────────────────┼──────────────┼─────────────────────┤
│ 5. 架构设计        │   AI 70%     │ 听 AI 建议，提问确认 │
│ 6. 技术栈          │   AI 80%     │ 让 AI 推荐，你选择   │
│ 7. 数据设计        │   AI 60%     │ 告诉 AI 要存什么     │
├────────────────────┼──────────────┼─────────────────────┤
│ 8. 代码实现        │   AI 100%    │ 完全交给 AI          │
│ 9. 测试验收        │   AI 80%     │ 你点点看对不对       │
│ 10. 部署上线       │   AI 90%     │ 跟着 AI 指导操作     │
│ 11. 运维监控       │   AI 90%     │ 出问题找 AI 帮忙     │
└────────────────────┴──────────────┴─────────────────────┘
```

### 简化版记忆口诀

```
上层靠你想（目标、需求、体验、设计）
中层一起商（架构、技术、数据）
下层 AI 扛（代码、测试、部署、运维）
```


---

## 🎨 设计师的优势

好消息！作为设计师，你已经具备了最重要的上层能力：

| 层级 | 你的现有能力 |
|------|-------------|
| ✅ 目标/愿景 | 你懂用户，懂市场需求 |
| ✅ 需求/功能 | 你做过产品设计，知道功能规划 |
| ✅ 用户体验 | 这是你的专业！用户流程、交互设计 |
| ✅ UI 设计 | 这更是你的强项！视觉、布局、风格 |

**AI 编程让你能"往下延伸"到技术层，而不需要从零学编程。**

## 💡 实用建议

### 1. 在开始之前，先回答这些问题

```
□ 这个项目是给谁用的？
□ 解决什么问题？
□ 核心功能是什么？（最多3个）
□ 不需要什么功能？
□ 用户的主要操作流程是什么？
```

### 2. 和 AI 沟通时的层级意识

| 当你说... | AI 理解的层级 |
|----------|--------------|
| "我想做一个记账 App" | 目标层 |
| "要能记录收入和支出" | 需求层 |
| "用户打开就能快速记一笔" | 体验层 |
| "用绿色代表收入，红色代表支出" | 设计层 |
| "数据要保存在手机里" | 数据层 |

### 3. 遇到问题时，往上层找原因

```
代码有 bug？      → 可能是需求没说清楚
功能不好用？      → 可能是体验流程没想好
做出来不满意？    → 可能是目标不明确
```

---

## 📝 本节小结

1. **项目有11个层级**：从目标到运维，越上层越重要
2. **改错代价**：上层错了代价大，下层错了容易改
3. **分工原则**：上层你主导，中层一起商量，下层 AI 负责
4. **设计师优势**：你已经具备最重要的上层能力
5. **核心心法**：先想清楚"做什么"，再让 AI 帮你"怎么做"
